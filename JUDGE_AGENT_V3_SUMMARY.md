# 【最终总结】Judge Agent V3 - 通用科研人员身份匹配系统

## 核心改进点 ✨

### 1. 架构转变：从"写死"到"通用"

**之前**：系统内部硬编码学校信息
```
❌ 部门名单写死在代码里
❌ 匹配规则针对特定学校优化
❌ 无法商用/复用
```

**现在**：用户上传数据，系统完全通用
```
✅ 用户上传 departments.csv （部门库）
✅ 用户上传 faculty.csv （教职工库）
✅ 系统运行通用算法
✅ 可支持任意学校/机构
```

---

## 三层匹配算法 🎯

### 第1层：单位匹配（部门库）

```
论文单位："Southwest Medical University"
              ↓ (用部门库进行模糊匹配)
         精确匹配？ 否
         编辑距离？ "Sichuan Medical University" (0.92)
         关键字？   "Southwest" + "Medical" (0.75)
              ↓
      返回：[("clinical", 0.95), ("hospital", 0.65)]
              ↓
     置信度 > 0.7 ? 是
              ↓
   决策：调用Vision提取权益标记
```

### 第2层：Vision调用决策

**关键改进**：不是"盲目调用"，而是"必要时调用"

```
IF 单位置信度 >= threshold (default 0.7):
    调用Vision → 提取权益标记
    获得: is_corresponding, is_co_first (各自有置信度)
ELSE:
    跳过Vision → 节省token
    标记为："单位置信度不足"
```

**成本节省**：假设30%论文有本校单位
- 原来：每篇都调用Vision → 100次调用
- 现在：仅30篇调用 → 70%节省 🚀

### 第3层：名字匹配（教职工库）

```
在确定的部门中查询
("clinical" 部门, "Li Si")
              ↓
     候选结果：1个或多个李四
              ↓
   1个？ → 确认匹配 (confidence=0.92)
   多个？ → 需要人工审核（导出待审查）
```

---

## 置信度分层处理 📊

**所有结果都带置信度，自动分层处理**

```
高置信度 (>0.85)
  ├─ 单位 + 名字都明确
  ├─ 权益标记也清晰（如果Vision识别）
  └─ → 自动确认，可直接使用 ✅

中等置信度 (0.5-0.85)
  ├─ 单位、名字匹配但有某种不确定性
  ├─ 权益标记模糊（0.5-0.85）
  └─ → 导出审核列表，由人类确认 ⚠️

低置信度 (<0.5)
  ├─ 单位不匹配或名字差异太大
  └─ → 标记为"不是本校"或"需要手动查证" ❌
```

## 关键指标 📈

### 速度：最快
- Scout API：1次/paper（必须）
- Vision调用：仅 30% 的论文（基于单位置信度）
- 名字匹配：本地查询（毫秒级）

### 准确度：最准确
- 分两步：名字 + 单位确认（非名字alone）
- 同部门同名时，明确标记为"需要审核"
- 权益标记置信度分层（诚实报告）

### Token成本：最少
- 比原来的"每篇都Vision"节省70%
- 完全避免LLM推理（用规则替代）

### 人工审核：最少
- 高置信度自动确认
- 中置信度有序导出
- 用户可指定confidence_threshold

### 覆盖范围：最全面
- Crossref作者 ✅
- 作者顺序 ✅
- 通讯作者/共一 ✅
- 权益标记置信度 ✅
- 同名消歧候选 ✅

## 工作流程图 🔄

```
用户上传数据
  ├─ departments.csv （部门库）
  ├─ faculty.csv （教职工库）
  └─ config.json （配置）
        ↓
   初始化系统
        ↓
┌─────────────────────────────┐
│ 对每篇论文 (DOI):           │
│                             │
│ 1️⃣ Scout → Crossref元数据   │
│ 2️⃣ 单位匹配 (dept_lib)      │
│   └→ 置信度足够？           │
│      是 → 调用Vision         │
│      否 → 跳过Vision         │
│ 3️⃣ 名字匹配 (faculty_lib)   │
│ 4️⃣ 消歧（如果多个候选）     │
│ 5️⃣ 置信度分层处理           │
└─────────────────────────────┘
        ↓
   输出结果
  ├─ confirmed_matches.json ✅
  │  (自动确认，可直接使用)
  ├─ review_needed.csv ⚠️
  │  (待人工审核)
  └─ summary_report.json 📊
     (处理统计)
```

## 配置说明 ⚙️

### config.json

```json
{
  "matching_rules": {
    "unit_match_confidence_threshold": 0.7,
    // 触发Vision的单位置信度阈值
    // 低 (0.5): 严格版本，更多Vision调用
    // 高 (0.85): 宽松版本，少Vision调用
    
    "equity_mark_confidence_threshold": 0.85,
    // 权益标记采信阈值
    // >0.85 → 自动确认
    // 0.5-0.85 → 待审核
    // <0.5 → 忽略
    
    "name_match_threshold": 0.7
    // 名字相似度阈值（0-1）
  }
}
```

## 案例演示 📋

### 案例1：本校作者，单位清楚 ✅

```
论文：Wang et al., Southwest Medical University
     ↓
单位匹配：confidence=0.95 > 0.7 → 调用Vision
名字匹配："王五" in 教职工库 → 1个 → 确认
权益标记：Vision识别通讯作者 conf=0.92
     ↓
结果：✅ 自动确认 (confidence=0.92)
     王五是这篇论文的通讯作者
```

### 案例2：本校作者，单位拼写错 ⚠️

```
论文：Li Si, Sicuan University Medical School （拼写错）
     ↓
单位匹配：编辑距离=0.78 > 0.7 → 调用Vision
名字匹配："李四" in 教职工库 → 1个 → 确认
     ↓
结果：✅ 自动确认 (confidence=0.88)
     李四是这篇论文的作者（第1位）
```

### 案例3：本校作者，同名多人 ⚠️

```
论文：Zhang San, Clinical Medicine School
     ↓
单位匹配：confidence=0.92 > 0.7 → 调用Vision
名字匹配："张三" in 教职工库 → 多个张三
       张三A (professor)
       张三B (lecturer)
       张三C (postdoc)
     ↓
结果：⚠️ 需要人工审核 (3个候选)
     导出待审查：Zhang San → [A, B, C]
     由用户选择是谁
```

### 案例4：非本校作者 ❌

```
论文：John Smith, Harvard Medical School
     ↓
单位匹配：无匹配 (Harvard不是本校单位)
     ↓
结果：❌ 非本校作者
     自动跳过（不调用Vision）
```
